## 加密傳輸流程（抗量子攻擊實現）

HTTP / WebSocket 主鏈路按 **Plan0 / Plan1 / Plan2** 分模式；Plan2 承擔 **ML-KEM-1024 + ML-DSA-87** 非對稱抗量子登錄與密鑰交換。**Plan0/1 登入態**對稱層為 **HMAC-SHA256 +（可選）AES-256-GCM**；**Plan2** 在 ML-KEM 協商後使用 **HKDF-SHA256 + HMAC-SHA256 + AES-256-GCM**。

### Plan 0: Base64 模式（登錄狀態，明文傳輸）

> **無 ML-DSA 外層籤**：僅 JWT 登錄態 + HMAC-SHA256；`e` 欄位為空。HTTP/WS 共用 Plan01 校驗邏輯（HTTP 以 `ctx.Path` 入規範串）。

```
客戶端                                服務端
  │                                    │
  │  1. 業務數據 JSON 序列化            │
  │     ↓                              │
  │  2. Base64 編碼 → d                │
  │     ↓                              │
  │  3. n = RandProtocolNonce() (32B)   │
  │     t = Unix 時間戳                 │
  │     ↓                              │
  │  4. HMAC-SHA256 → s            │
  │     HMAC-SHA256(path+d+n+t+p+u, TokenSecret)
  │     ↓                              │
  ├─────────  發送請求  ────────────────▶│
  │    {d, t, n, p:0, s}  (無 e)       │
  │                                    │  5. 校驗 n 長度 & 時間窗
  │                                    │  6. 驗證 HMAC (s)
  │                                    │  7. 重放檢測 (s / n)
  │                                    │  8. Base64 解碼 d
  │                                    │  9. 業務處理 → 響應 HMAC
  │◀────────  返迴響應  ─────────────────┤
  │    {c, m, d, t, n, p:0, s}         │
  │                                    │
10. 驗證響應 HMAC & 解碼                │
```

### Plan 1: AES-GCM 模式（登錄狀態，對稱加密）

```
客戶端                                服務端
  │                                    │
  │  1. 業務數據 JSON 序列化            │
  │     ↓                              │
  │  2. 獲取 Token Secret              │
  │     (動態生成，不存儲)              │
  │     ↓                              │
  │  3. n = RandProtocolNonce(); t      │
  │     ↓                              │
  │  4. AES-256-GCM 加密 → d           │
  │     Key: TokenSecret[:32]          │
  │     AAD: t+n+p+path (協議 n 入 AAD) │
  │     (GCM IV 在密文內，12 字節)      │
  │     ↓                              │
  │  5. HMAC-SHA256 → s            │
  │     ↓                              │
  ├─────────  發送請求  ────────────────▶│
  │    {d, t, n, p:1, s}  (無 e)       │
  │                                    │  6. JWT → GetTokenSecret
  │                                    │  7. 驗證 HMAC & 時間窗 & n
  │                                    │  8. AES-GCM 解密 (AAD 綁定)
  │                                    │  9. 業務處理 → 響應 GCM+HMAC
  │◀────────  返迴響應  ─────────────────┤
  │                                    │
10. 驗證 HMAC & GCM 解密                │
  │                                    │
16. 業務數據處理                       │
```

### Plan 2: ML-KEM + AES-GCM + ML-DSA-87 模式（匿名狀態，混合加密 + 雙向籤名）

```
客戶端                                服務端
  │                                    │
  │  1. 請求服務端封裝公鑰              │
  ├─────────  POST /key  ─────────────▶│
  │◀────── 服務端 ML-KEM 封裝公鑰 ek ──┤
  │    PublicKey { key: ek_b64, noc, exp, sig } │
  │                                    │
  │  2. 驗證服務端 ML-DSA 外層籤名        │
  │     Verify(Valid, SHA256(key+noc+exp))│
  │     使用客戶端預設的服務端 ML-DSA 公鑰 │
  │                                    │
  │  3. ML-KEM 封裝（單向）             │
  │     Encapsulate(server_ek)         │
  │     → shared_raw, kem_ct           │
  │     ↓                              │
  │  4. HKDF-SHA256 密鑰派生（node.HKDFKey）│
  │     Key = HKDF-SHA256(shared_raw, noc)│
  │     ↓                              │
  │  5. 業務數據 JSON 序列化            │
  │     ↓                              │
  │  6. AES-256-GCM 加密               │
  │     ↓                              │
  │  7. HMAC-SHA256 → s 欄位       │
  │     ↓                              │
  │  8. ML-DSA-87 外層籤名 → e 欄位     │
  │     Sign(SHA256(path+d+n+t+p+u))   │
  │     ↓                              │
  ├─────────  發送業務請求  ───────────▶│
  │    Authorization: {                │
  │      key: server_ek,               │
  │      tag: kem_ct_b64,              │
  │      noc, sig, exp                 │
  │    }                               │
  │    JsonBody { d, t, n, p:2, s, e }   │
  │                                    │
  │  9. 驗證客戶端 ML-DSA 外層籤名        │
  │ 10. Decapsulate(dk, kem_ct)→shared │
  │ 11. HKDF-SHA256 → sharedKey        │
  │ 12. 驗證 HMAC、時間戳、Nonce         │

### WebSocket 服務端推送（`code=300`）

與業務報文 **同一 MAC 算法**（`SignBodyMessage`）；僅 **密鑰** 不同（廣播密鑰，非 JWT Secret）：

```
服務端 SendToSubject                     客戶端 SDK
  │                                        │
  │  n = RandProtocolNonce()               │
  │  s = SignBodyMessage(                  │
  │      path+d+n+t+p, broadcastKey)       │
  │  JsonResp { c:300, r, d, n, t, p, s }  │
  ├───────────────────────────────────────▶│
  │                                        │ verifyPushMessageSignature
  │                                        │ SetBroadcastKey 須與
  │                                        │ PushKeyProvider 一致
  │                                        │ → 解密 d → 訂閱 Handler
```

   │   │ Payload:                        │
   │   │   sub, dev, exp, jti, ...       │
   │   │                                 │
   │   │ Signature (第三段):             │
   │   │   part = b64(header).b64(payload)│
   │   │   sig = B64( HMAC-SHA256(       │
   │   │     key=TokenKey,               │
   │   │     msg=VerifyType|SEP|token,   │
   │   │     見 GetTokenSecretExtract )  │
   │   └─────────────────────────────────┘
   │
   ├─▶ 動態生成會話密鑰（GetTokenSecret）
   │   └─▶ HMAC-SHA256(key=TokenKey, msg=SecretType|SEP|完整JWT)
   │       • 供 HTTP/WS 業務 HMAC-SHA256 / AES-GCM
   │       • 每次按需計算、不長期存儲
   │
   └─▶ 返回客戶端
       {
         token: "eyJhbGc...",
         secret: "dynamic_secret",
         expired: 1698209856
       }

2. 客戶端存儲
   │
   ├─▶ 安全存儲 Token + Secret
   │   • Web: localStorage/sessionStorage
   │   • App: KeyChain/KeyStore
   │   • 不在 Cookie 中存儲 (防 CSRF)
   │
   └─▶ 後續請求攜帶
       Header: Authorization: token
       Body: AES-GCM 加密 (使用 secret)

3. 服務端驗證
   │
   ├─▶ SessionFilter 攔截
   │   │
   │   ├─▶ 提取 Token
   │   │   Header: Authorization
   │   │
   │   ├─▶ 驗證 Token 第三段（Subject.Verify）
   │   │   • 復算 HMAC-SHA256，與 part3 比較
   │   │
   │   ├─▶ 驗證 Token 過期
   │   │   • exp < 當前時間 → 拒絕
   │   │
   │   ├─▶ 提取 Payload
   │   │   • sub (用戶 ID)
   │   │   • dev (設備類型)
   │   │   • 自定義欄位
   │   │
   │   ├─▶ 動態生成 Secret
   │   │   GetTokenSecret(token, tokenKey)
   │   │   (與登錄時相同算法)
   │   │
   │   └─▶ 注入 Context
   │       ctx.Subject = Subject{
   │         Token: token,
   │         Payload: payload,
   │         Secret: secret
   │       }
   │
   ├─▶ RoleFilter 權限驗證
   │   │
   │   ├─▶ 讀取 RBAC 配置
   │   │   routerConfig.Permission
   │   │
   │   ├─▶ 驗證登錄狀態
   │   │   NeedLogin → 檢查 Token
   │   │
   │   ├─▶ 驗證角色權限
   │   │   • HasRole: 用戶實際角色
   │   │   • NeedRole: 接口所需角色
   │   │   • MatchAll: 全匹配/部分匹配
   │   │
   │   └─▶ 權限決策
   │       ✅ 通過 → 繼續執行
   │       ❌ 拒絕 → 返回 403
   │
   └─▶ 業務邏輯
       • 獲取用戶信息: ctx.Subject.GetUserID()
       • 獲取設備類型: ctx.Subject.GetDev()
       • 解密請求數據: ctx.GetTokenSecret()

4. Token 註銷
   │
   ├─▶ 調用註銷接口
   │   POST /logout
   │
   ├─▶ 可選: 黑名單機制
   │   • Redis 存儲 Token Hash
   │   • TTL = Token 剩餘有效期
   │   • 驗證時檢查黑名單
   │
   └─▶ 客戶端清除本地存儲
```
